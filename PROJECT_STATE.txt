V1.8 CHANGES — RATED OPPONENT LADDER + PERSONAS + BANTER
  (verified: chess 26/26; rating 20/20; node --check on every touched file; a
  poisoned-capture sanity probe confirms the strength gradient — the two depth-1
  personas hang a queen-to-recapture ~55% of the time, every depth-2+ persona 0%;
  setup-screen + board headless screenshots clean; load = zero app console errors)
  - AI IS NOW PROFILE-DRIVEN (js/ai.js): the hardcoded 3-row DIFFS table became a
    normProfile() that accepts EITHER a persona profile object
    {depth, blunder, noise, nodeCap?} OR a legacy numeric level (0/1/2, kept in a
    small LEGACY table for back-compat). Strength is achieved by DELIBERATE
    weakening, not a strong engine throwing games:
      * depth   — plies of full-width search (1 = barely tactical).
      * blunder — flat per-move chance to abandon the best move and play a RANDOM
        other legal move (this is what makes weak personas hang pieces).
      * noise   — eval jitter (the old "jitter") so non-blundering play feels human.
      * quiescence is derived (depth >= 3), so shallow personas never see the
        recapture and genuinely hang material; deep ones do and don't.
    chooseMove/chooseMoveAsync now take `profile` (object or number); forced mate
    is never randomised away (mate scores sit far outside the noise band).
  - ROSTER (js/opponents.js, NEW — MG.Opponents): 10 named, orchestra-themed
    conductor personas, weakest→strongest, each
    { id, name, blurb, rating, klass, depth, blunder, noise, tint, lines{win,
    bigCapture, lose} }. Ratings span the FIDE/USCF class bands via classOf():
      Piccolo Pim 820 (Novice) · Triangle Tina 1010 / Reed Wobble 1180 (Class E) ·
      Viola Vance 1300 (Class D) · Cornet Cleo 1450 / Tempo Tan 1560 (Class C) ·
      Fugue Fennimore 1680 (Class B) · Crescendo Cruz 1820 / Diva Della 1930
      (Class A) · Maestro Magnus 2010 (Expert). depth ramps 1→4, blunder .45→0,
      noise 140→0. DEFAULT_ID='tempo'. get/has/byClass() helpers; pure data, no
      DOM, require()-able. `rating` is the FIXED opponent rating fed to the rating
      system (replaces the old AI_RATINGS=[1000,1500,2000] mapping; that constant
      still exists in rating.js, now unused by main.js).
    HONESTY: the ladder STOPS at the engine's honest ceiling (~2010, labelled
    Expert). No 2400+/2700 personas that secretly play weak. True master strength
    needs an opening book + transposition table + a richer eval (see backlog).
  - OPPONENT PICKER (index.html #opp-picker, css .opp-*, ui.js buildOpponentPicker):
    the 3-button #seg-diff segment was REPLACED by a scrollable card list grouped by
    class band. Each card = a tinted procedural conductor portrait (drawIcon('K','b')

  - WIRING (js/main.js): session gained {opponent (the persona), aiProfile}; level

V1.7 CHANGES — PLAYER PROFILES + CHESS RATING SYSTEM
  (verified: chess 26/26; rating 20/20; node --check on all touched files; a
  headless browser-DOM probe drove the full profile lifecycle 9/9 — create,
  Elo+Glicko game updates, system-switch seeding, localStorage round-trip,
  delete→Guest; title + profiles screenshots clean; load = zero console errors)
  - RATING MATH (js/rating.js, NEW — MG.Rating): pure, DOM-free, node-testable.
      * Elo (DEFAULT): expected(R,Ropp)=1/(1+10^((Ropp-R)/400)); updateElo applies
        R'=R+K(S-E). kFactor: 40 provisional (<30 games) / 20 / 10 (>=2400). New
        profiles start at 1200. AI_RATINGS=[1000,1500,2000] (Student/Performer/
        Virtuoso) are the fixed opponent ratings for vs-CPU games.
      * Glicko-2 (FULL Glickman-2013 algorithm): g(), E, v, Δ, the volatility
        equation solved by Illinois, then φ*/φ'/μ'. updateGlicko(rating,rd,vol,
        results,τ) takes a rating period (we pass one game/period live); also
        handles an idle period (RD grows). SCALE=400/ln(10)=173.7178; defaults
        RD 350 / σ 0.06 / τ 0.5. Pinned to the published worked example
        (1464.06 / 151.52 / 0.05999) in tests/test_rating.js.
      * Display equivalents are ESTIMATES ONLY (different models, not unit
        conversions): eloToUSCF=Elo+75, eloToECF=(Elo-700)/7.5, estimateLine()
        for the Options "USCF ≈ … · ECF ≈ …" readout.
      * seedFor(system,current): switching the computing system carries the
        current number across (Glicko shares the Elo scale) and resets Glicko
        RD/σ. SYSTEMS = {elo:'Elo', glicko:'Glicko-2'}.
  - PROFILE PERSISTENCE (js/profiles.js, NEW — MG.Profiles): own localStorage key
    'mg_profiles' = {profiles:{id:{...}}, activeId}. A profile is {id,name,system,
    rating,rd,vol,games,wins,draws,losses,history[]} (history capped at 50).
    "Guest" is a SYNTHETIC, never-saved profile (id 'guest') that tracks nothing.
    API: load/save, all/get/active/isGuest, setActive/playAsGuest, create(name,
    system), remove, setSystem(prof,sys) [seeds via MG.Rating], recordGame(prof,
    score,oppRating,opts) → {before,after,delta,system} (null for Guest), and
    ratingLabel(). recordGame routes to Elo or Glicko per the profile's system.
  - UI (index.html, css/style.css, js/ui.js):
      * PROFILE PICKER: a pill "profile-chip" on the TITLE screen (top) and SETUP
        screen ("Player" group) both open a new SCREEN-PROFILES; a "Profiles"
        menu button does too. The profiles screen lists Guest + every saved
        profile (name, system, W/D/L, rating; active highlighted), each selectable
        with a ✕ delete (confirm dialog); a name input + Create; "Play as Guest".
      * OPTIONS gains a "Rating System" segmented control (Elo / Glicko-2) that
        re-seeds the ACTIVE profile (no-op for Guest) plus a live "USCF ≈ … ·
        ECF ≈ …" estimate line (#rating-estimate). seg-rating reflects the active
        profile's system whenever Options opens (refreshRatingOption()).
      * HUD chip (#hud-profile, top-left) shows "Elo 1234" during play, hidden for
        Guest. GAME-OVER card gained #go-rating: "Elo 1200 → 1212 (+12)" coloured
        green/red by delta. refreshProfileUI() keeps every surface in sync.
      * ui.js: SCREENS += 'screen-profiles'; MG.Profiles.load() in init; helpers
        openProfiles/createProfileFromInput/renderProfiles/refreshProfileUI/
        refreshRatingOption/setHudProfile; showGameOver gained an optional 3rd
        ratingHtml arg.
  - WIRING (js/main.js): startGame resets this.ratedThisGame and calls
    MG.UI.setHudProfile(). endGame computes the result via applyRatingResult(winner)
    BEFORE the 900ms card delay and passes the rating line into showGameOver.
    applyRatingResult: skips 2P-local and Guest; score = 0.5 (draw, winner null) /
    1 / 0 from the human's view; oppRating = AI_RATINGS[level] for cpu, or the
    player's OWN rating for online (honour system, unknown opponent → graded as an
    equal). Guarded by ratedThisGame so a game counts once. onPeerLeft (disconnect)
    deliberately does NOT rate — the game was incomplete. New dev hook
    ?screen=profiles opens the profiles screen.
  - SCRIPT ORDER: rating.js then profiles.js load AFTER board.js and BEFORE ui.js
    (ui.js uses MG.Profiles; profiles.js uses MG.Rating).
  - NOT TOUCHED: chess.js/ai.js/battle.js/sprites.js/fx.js/board.js/net.js/audio.js
    unchanged — zero risk to the verified engine + choreography (soak not re-run by
    design, as none of those files changed).

RATING SYSTEM — DONE in v1.7 (original plan)
  STATUS: IMPLEMENTED — see V1.7 CHANGES at the top. The recommended design below
  was followed essentially as written (Elo default + full Glicko-2, honest
  seeding/estimates, profile persistence, profile UI, endGame wiring, tests).
  Kept here for the rationale.

  WHY DEFERRED: a rating that "globally saves as you play" needs more than a
  formula — it needs a player-profile + persistence layer (create/select/resume a
  profile, stored in localStorage; cleared if the user wipes site data), rating
  math, profile UI on the title/setup screens, and wiring rating updates into
  game-end. That's a self-contained feature best done in its own pass, not bolted
  onto this one. (Research source: en.wikipedia.org/wiki/Chess_rating_system.)

  RECOMMENDED DESIGN:
   - PRIMARY SYSTEM = Elo (the single most widely used; FIDE & USCF since the
     1960s–70s, simple and transparent). Per game:
         E = 1 / (1 + 10^((Ropp - R)/400))        (expected score)
         R' = R + K * (S - E)                      (S = 1 win / 0.5 draw / 0 loss)
     Provisional K=40 (first ~30 games), K=20 above, K=10 for high ratings.
     New profiles start at 1200. For vs-CPU games, give each AI level a fixed
     "opponent rating" (e.g. Student ~1000, Performer ~1500, Virtuoso ~2000) so a
     human's rating converges sensibly.
   - "SELECT A DIFFERENT SYSTEM" — important nuance to capture in the prompt: Elo,
     Glicko-2 (chess.com/lichess), USCF, ECF, DWZ are DIFFERENT statistical models,
     not unit conversions, so there's no exact two-way conversion. The realistic,
     honest feature is: let the player choose which system COMPUTES their rating,
     and when switching, SEED the new system from their current number via the
     standard published linear approximations, then let it evolve on its own:
         USCF ≈ FIDE + 50..100      ECF = (Elo - 700) / 7.5   (Elo = 7.5*ECF + 700)
     Glicko-2 uses the same ~1500-centred scale as Elo, plus a rating deviation
     (RD, start 350) and volatility (σ, start 0.06); seed rating ≈ current Elo.
     Show the converted value as an estimate, clearly labelled, when switching.
   - OPTIONAL SECOND ENGINE = Glicko-2 (better for few games / inactivity). Only
     add if the player picks it; its full update step is ~30 lines (compute g(RD),
     E, v, Δ, then iterate the volatility equation). Keep Elo as the default.
   - PERSISTENCE: a `profiles` map + `activeProfile` id in mg_prefs (or a separate
     localStorage key 'mg_profiles'). Each profile: {name, system, rating, rd, vol,
     games, wins, draws, losses, history[]}. Title/Setup gets a small profile
     picker (create / pick / "Guest" = no rating). Update on endGame for cpu (and
     optionally online, honour-system) — NOT for 2p-local (whose rating?).
   - TESTS: add tests/test_rating.js (Elo expected-score + a known Glicko-2 step
     from the Glickman worked example) so the math is pinned like the chess engine.

  MODEL / EFFORT for that pass: Opus 4.8 at HIGH effort (exact math + persistence +
  UI + tests warrant the thoroughness; it's not a quick one). It is self-contained
  and won't touch chess.js, so risk to the verified engine is low.
