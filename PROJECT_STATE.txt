    MG.drawPiece2D): each piece is a single CLOSED silhouette traced in a
    resolution-independent fraction space (x = fraction of piece height either side of

    explicit asymmetric horse-head profile. Small accents: a knight eye + a bishop
    slit (in the per-colour detail colour). Per-piece heights are a fraction of the
    tile (P .60 ... K .80). All shapes authored from scratch here — NO copied artwork
    or third-party chess font, so the commercial-original guarantee holds (no glyph
    licence needed; nothing loaded from a font for the pieces).
  - NOT TOUCHED: every other js file. No new <script> (all additions live in the
    existing board.js). No HTML/CSS change (the View button already existed).

  (A) OPENING BOOK — js/opening_book.js (NEW, MG.OpeningBook; pure + DOM-free,
      require()-able; loads AFTER chess.js, before pgn.js):
    - DATA: LINES = ~20 hand-authored mainlines (Ruy López/Italian/Scotch/Petrov/
      Two Knights · Sicilian Najdorf+Classical · French · Caro-Kann · Scandinavian ·
      QGD/QGA/Slav · KID/Nimzo/QID · London · English · Réti), a handful of plies
      deep, covering both 1.e4 and 1.d4 plus flank openings and the common replies.
      Opening THEORY isn't copyrightable and these lines are authored here move by
      move — NOT copied from a specific published book file; names are descriptive.
    - BUILD: replaying each line from the start records, at every position, which SAN
      was played + how many lines pass through it (its weight). BOOK is keyed by the
      engine's posKey() string, so transpositions merge automatically (141 distinct
      book positions). Built lazily on first use; a SAN the engine won't accept is
      console.warn'd and that line is truncated (a bad token can never wedge the book
      or surface an illegal move). candidates(game) re-matches each stored SAN against
      the live legalMoves(), so an offered move is always legal in its position.
    - PERSONA POLICY (lives in the book so callers stay simple): bookChance(profile)
      derives a per-persona probability from the persona's RATING (≈1100 Elo→0,
      ≈2000→1; falls back to depth, and legacy numeric levels map 0/1/2→.12/.55/.9),
      scaled to ≈.08 (weakest) … ≈.96 (strongest). pickFor(game, profile, rng?)
      returns a weighted-random book move with that probability, else null. So strong
      personas open from book often (variety + perceived strength) and weak ones
      rarely — the ladder gradient is preserved (a novice can't lean on theory it
      hasn't "learned").
    - WIRING: consulted at the TOP of MG.AI.chooseMove (so the live cpuMove path +
      its thinking-delay get it for free via chooseMoveAsync; main.js needed NO
      change). Guarded by `MG.OpeningBook &&` so ai.js still runs standalone under
      Node (test_chess.js doesn't load the book → book path inert there). Puzzle
      defence is scripted, and puzzle FENs aren't openings, so the book never
      interferes.

  (B) ZOBRIST HASHING — js/chess.js (ADDITIVE; does NOT affect move-gen/legality):
    - A fixed-seed xorshift PRNG builds the Zobrist tables once (12 piece×64 squares,
      4 castle rights, 8 ep files, 1 side key). The position hash is two signed
      32-bit halves hashHi/hashLo. computeHash() rebuilds from scratch (used at
      reset()/loadFEN() and as an authoritative resync at the search root). During
      search the hash is maintained INCREMENTALLY in _apply (piece leaves `from`,
      capture/ep-victim leaves, piece arrives — promotion changes type, castling rook
      hop, any flipped castling right, ep-file in/out, side flip) and RESTORED
      VERBATIM in _unapply from a snapshot stashed in the undo record (so no inverse
      XOR maths, and _unapply can't drift). Verified: incremental == from-scratch at
      every node of the perft(3) tree, and exact restore after _unapply. Repetition
      detection STILL uses the string posKey() (proven); Zobrist is consumed only by
      the AI's TT, and the hash deliberately ignores the move clocks so transpositions
      collide (which is what a TT wants).

  (C) TRANSPOSITION TABLE + ITERATIVE DEEPENING — js/ai.js (negamax/chooseMove):
    - TT: a per-search Map<hashLo,{hi,depth,score,flag,fromTo}> with full 64-bit
      verification (entry.hi must match). Standard fail-soft bounds (EXACT/LOWER/
      UPPER) give cutoffs when entry.depth>=needed; the stored best move (packed
      from<<8|to) is pulled to the front of the move list for sharper pruning. MATE
      scores (|score|>=MATE_GUARD=90000) are NEVER trusted from the TT (they're
      ply-distance sensitive) and over-budget nodes aren't stored — so the
      "never randomise away a forced mate" guarantee and the fail-hard mate-score
      history are both safe.
    - QUIESCENCE is UNCHANGED and deliberately TT-FREE — it keeps its fail-soft
      "always return real evaluations" behaviour exactly as PROJECT_STATE warns.
    - ITERATIVE DEEPENING: chooseMove now searches depth 1..cfg.depth, reusing the TT
      and seeding each pass's root order from the previous pass (principal-variation-
      first). A persona still searches EXACTLY cfg.depth plies — nominal depth, and
      thus rated strength, is unchanged — it just reaches that depth far more cheaply
      (≈40-60% fewer nodes on opening positions). A pass that blows the node cap is
      discarded so the last CLEAN pass is used; depth-1 personas (ID off, single
      pass) are byte-for-byte the old behaviour, so the weak end of the ladder is
      untouched. The historic root window + the blunder/noise/mate-shortcut weakening
      tail are preserved verbatim, so persona character is identical.
    - A/B HOOK: chooseMove(game, profile, opts) gained an opts arg ({tt,id,book}, all
      default on) used ONLY by the strength harness to toggle a feature off and
      measure its contribution. The game never passes opts.

  STRENGTH MEASUREMENT (tests/strength_probe.js, NEW — informational, not a gate):
    - POISONED-CAPTURE GRADIENT (a position where Qxd5 wins a pawn but hangs the
      queen to a recapture): depth-1 personas grab it 24%/41% of the time, every
      depth-2+ persona ~0-1%, and the no-blunder summit (Della/Magnus) NEVER —
      the strength gradient + the honesty invariant both hold after the changes.
    - BUDGET A/B: the new search (TT+ID) vs the OLD one (single fixed-depth pass) at
      the SAME 12k-node cap over 30 random-opening games → NEW 30-0 (100%). The TT+ID
      genuinely converts a node budget into stronger play (the point of the pass).
    - DEPTH A/B (informational, used for the persona decision): depth-5 vs depth-4
      (both TT+ID) → depth-5 ~69%. Deeper IS stronger in self-play, BUT both share
      this engine's crude-eval blind spots, so this overstates real-world strength.
      Combined with the still-missing richer evaluation, this does NOT honestly
      support a >2010 rung — so none was added (see backlog).

RATING SYSTEM — DONE in v1.7 (original plan)
  STATUS: IMPLEMENTED — see V1.7 CHANGES at the top. The recommended design below
  was followed essentially as written (Elo default + full Glicko-2, honest
  seeding/estimates, profile persistence, profile UI, endGame wiring, tests).
  Kept here for the rationale.

  WHY DEFERRED: a rating that "globally saves as you play" needs more than a
  formula — it needs a player-profile + persistence layer (create/select/resume a
  profile, stored in localStorage; cleared if the user wipes site data), rating
  math, profile UI on the title/setup screens, and wiring rating updates into
  game-end. That's a self-contained feature best done in its own pass, not bolted
  onto this one. (Research source: en.wikipedia.org/wiki/Chess_rating_system.)

  RECOMMENDED DESIGN:
   - PRIMARY SYSTEM = Elo (the single most widely used; FIDE & USCF since the
     1960s–70s, simple and transparent). Per game:
         E = 1 / (1 + 10^((Ropp - R)/400))        (expected score)
         R' = R + K * (S - E)                      (S = 1 win / 0.5 draw / 0 loss)
     Provisional K=40 (first ~30 games), K=20 above, K=10 for high ratings.
     New profiles start at 1200. For vs-CPU games, give each AI level a fixed
     "opponent rating" (e.g. Student ~1000, Performer ~1500, Virtuoso ~2000) so a
     human's rating converges sensibly.
   - "SELECT A DIFFERENT SYSTEM" — important nuance to capture in the prompt: Elo,
     Glicko-2 (chess.com/lichess), USCF, ECF, DWZ are DIFFERENT statistical models,
     not unit conversions, so there's no exact two-way conversion. The realistic,
     honest feature is: let the player choose which system COMPUTES their rating,
     and when switching, SEED the new system from their current number via the
     standard published linear approximations, then let it evolve on its own:
         USCF ≈ FIDE + 50..100      ECF = (Elo - 700) / 7.5   (Elo = 7.5*ECF + 700)
     Glicko-2 uses the same ~1500-centred scale as Elo, plus a rating deviation
     (RD, start 350) and volatility (σ, start 0.06); seed rating ≈ current Elo.
     Show the converted value as an estimate, clearly labelled, when switching.
   - OPTIONAL SECOND ENGINE = Glicko-2 (better for few games / inactivity). Only
     add if the player picks it; its full update step is ~30 lines (compute g(RD),
     E, v, Δ, then iterate the volatility equation). Keep Elo as the default.
   - PERSISTENCE: a `profiles` map + `activeProfile` id in mg_prefs (or a separate
     localStorage key 'mg_profiles'). Each profile: {name, system, rating, rd, vol,
     games, wins, draws, losses, history[]}. Title/Setup gets a small profile
     picker (create / pick / "Guest" = no rating). Update on endGame for cpu (and
     optionally online, honour-system) — NOT for 2p-local (whose rating?).
   - TESTS: add tests/test_rating.js (Elo expected-score + a known Glicko-2 step
     from the Glickman worked example) so the math is pinned like the chess engine.

  MODEL / EFFORT for that pass: Opus 4.8 at HIGH effort (exact math + persistence +
  UI + tests warrant the thoroughness; it's not a quick one). It is self-contained
  and won't touch chess.js, so risk to the verified engine is low.
